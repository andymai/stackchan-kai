# `stackchan-tts` assets

Baked verbal-phrase PCM committed alongside the source.

## Layout

```
assets/
├── manifest.toml          # Source-of-truth: phrase text per locale
├── en/
│   ├── greeting.pcm
│   ├── acknowledge_name.pcm
│   └── battery_low.pcm
└── ja/
    ├── greeting.pcm
    ├── acknowledge_name.pcm
    └── battery_low.pcm
```

## Format

Raw 16 kHz / 16-bit signed / mono / little-endian PCM. No header (no WAV, no Opus).

Matches the firmware audio task's I²S configuration so the `BakedBackend` can `include_bytes!` directly into a `&'static [i16]` and push samples without runtime decode. Trade-off: ~32 KB/s — fine for short phrases (a 2-second phrase is 64 KB; the v1 catalog is well under 1 MB).

## Workflow

Phrases are regenerated by `just bake-tts`, which runs [Piper](https://github.com/rhasspy/piper) under `scripts/bake-tts.sh`:

```bash
just bake-tts                       # all locales × all phrases
just bake-tts en greeting           # one specific phrase
```

Generated `.pcm` files are committed to the repo so fresh checkouts and CI builds don't depend on a TTS binary. Update the manifest, run the recipe, commit both together.

## Stub PCM

Initial commits ship silent-PCM stubs (zeros) so `include_bytes!` succeeds and the firmware build is end-to-end runnable. The verbal phrases play as silence until the user runs `just bake-tts`. The diff after baking shows real samples replacing zeros — cleanly visible in code review.
